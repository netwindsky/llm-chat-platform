# llama-server 配置参数详解（针对 Qwen3.5-122B MoE 模型优化）

本配置基于以下硬件环境：
- **GPU**: NVIDIA GeForce RTX 4090 (24GB 显存)
- **CPU**: Intel i9-14900KF (8 P-core + 16 E-core, 共 32 线程)
- **内存**: 96GB DDR5 5600MT/s 双通道
- **模型**: Qwen3.5-122B-A10B-UD-Q4_K_XL (MoE 架构, 总参数量 122B, 激活 10B, 量化 4-bit, 文件约 70GB)

经过多轮调试，最终获得稳定 **11 token/s** 的生成速度。以下逐一解释命令中每个参数的作用及调试思路。

```bash
runtimes\llama\bin\llama-server.exe ^
    -m %MODEL_PATH% ^
    --host 127.0.0.1 ^
    --port 18081 ^
    -c 8192 ^
    -ngl 48 ^
    -ncmoe 48 ^
    -t 12 ^
    --cpu-range 0-11 ^
    -b 1024 ^
    -ub 1024 ^
    --cont-batching ^
    --cache-type-k q4_0 ^
    --cache-type-v q4_0 ^
    -fa on
```

---

## 1. 基础运行参数

### `-m %MODEL_PATH%`
- **作用**：指定要加载的 GGUF 模型文件路径。`%MODEL_PATH%` 是环境变量，实际使用时替换为绝对路径，例如 `models\Qwen3.5-122B-A10B-UD-Q4_K_XL.gguf`。
- **调试原因**：明确指向正确的模型文件。

### `--host 127.0.0.1`
- **作用**：监听本地回环地址，只允许本机访问服务。
- **调试原因**：保证安全性，避免外部网络访问；同时减少网络开销，适用于本地推理。

### `--port 18081`
- **作用**：指定服务监听的端口号。
- **调试原因**：选择一个未被占用的端口，便于通过 API 调用（如 OpenAI 兼容接口）。

---

## 2. 上下文管理

### `-c 8192`
- **作用**：设置上下文窗口大小为 8192 tokens。
- **调试原因**：
  - 模型原生支持极长上下文（262k），但过大的上下文会显著增加 KV 缓存开销，占用显存/内存并降低速度。
  - 8192 是平衡点：足够覆盖绝大多数对话历史，同时 KV 缓存量化为 4-bit 后仅占用约 54MB 显存，对性能影响极小。
  - 若需要处理超长文档，可适当提高此值，但需注意显存变化。

---

## 3. GPU/CPU 卸载策略（针对 MoE 架构的核心优化）

### `-ngl 48`
- **作用**：将模型的 **前 48 层** 卸载到 GPU 处理。
- **调试原因**：
  - Qwen3.5-122B 共有 48 层。设置 `-ngl 48` 意味着所有层的**非专家部分**（如注意力层、LayerNorm 等）都将由 GPU 计算。
  - 非专家层占模型总参数量的一小部分，卸载全部 48 层后，GPU 显存占用仅约 **5.5GB**（远低于 24GB 上限），为 KV 缓存和其他开销留足空间。
  - 让 GPU 负责注意力机制等并行度高的运算，能极大提升推理速度。

### `-ncmoe 48`
- **作用**：将 **所有 48 层中的 MoE 专家层权重** 保留在 CPU 内存中，由 CPU 负责计算。
- **调试原因**：
  - 这是 MoE 模型混合推理的关键参数。专家层占总参数量的 90% 以上（约 70GB），显存放不下。
  - 强制专家层留在 CPU，GPU 只处理非专家层，既避免显存溢出，又让 GPU 专注于它擅长的计算。
  - 参数值 48 表示对全部 48 层生效，确保所有专家权重都不进入显存。

**组合效果**：`-ngl 48` + `-ncmoe 48` 实现了 GPU 与 CPU 的分工协作——GPU 负责注意力等密集计算，CPU 负责稀疏的专家网络，充分利用了 4090 的算力和 96GB 的大内存。

---

## 4. CPU 线程控制（针对 i9-14900KF 混合架构）

### `-t 12`
- **作用**：使用 12 个 CPU 线程进行推理。
- **调试原因**：
  - i9-14900KF 拥有 8 个性能核（P-core，支持超线程）和 16 个能效核（E-core）。
  - 在内存带宽敏感的 MoE 推理中，过多线程可能导致资源争用，反而降低效率。
  - 经过测试，`-t 8`（纯 P-core 物理核心）得到约 10.98 t/s，`-t 16`（P-core + 超线程）约 10.98 t/s，而 `-t 12` 达到 **11.04 t/s**，略优。
  - 12 个线程恰好覆盖了 8 个 P-core 中的 4 个超线程 + 部分 P-core 物理核心？实际上 `--cpu-range 0-11` 绑定了前 12 个逻辑核心（即 6 个 P-core 的超线程），具体分配由操作系统调度。此配置在实测中表现最佳。

### `--cpu-range 0-11`
- **作用**：将 CPU 线程绑定到逻辑核心 0 至 11。
- **调试原因**：
  - 与 `-t 12` 配合，强制线程只运行在指定的核心上，避免调度到慢速的 E-core 上。
  - 0-11 通常是前 6 个 P-core 的超线程（逻辑核心 0-5 对应物理核心 0-5 的线程 0，6-11 对应相同物理核心的线程 1），这样既利用了超线程的轻度并行，又防止线程飘移到 E-core。
  - 绑定核心能减少上下文切换，提高缓存命中率，尤其对内存带宽敏感的任务有利。

---

## 5. 批处理与调度优化

### `-b 1024` 与 `-ub 1024`
- **作用**：设置物理批大小（`-b`）和最大批大小（`-ub`）均为 1024 tokens。
- **调试原因**：
  - 较大的批处理可以让 GPU 和 CPU 一次性处理更多 token，提高吞吐量。
  - 1024 是经过测试的平衡点：既能充分利用计算资源，又不会因批过大导致首 token 延迟过高。
  - 曾尝试 `-b 2048`，但速度提升不明显，且可能增加内存压力，故保留 1024。

### `--cont-batching`
- **作用**：启用连续批处理（continuous batching），允许动态地将多个请求合并处理。
- **调试原因**：
  - 当多个请求并发时，此选项能显著提高吞吐量，减少排队延迟。
  - 即使在单用户场景下，开启也无负面影响，保持默认启用。

---

## 6. KV 缓存量化

### `--cache-type-k q4_0` 和 `--cache-type-v q4_0`
- **作用**：将 KV 缓存（key 和 value 缓存）量化为 4-bit（q4_0 格式）。
- **调试原因**：
  - KV 缓存占用大量显存，尤其当上下文较长时。量化后可将缓存大小压缩数倍。
  - 设置为 q4_0 后，8192 上下文的 KV 缓存仅占用 **54MB**，几乎可以忽略不计。
  - 量化对精度影响极小，但显著降低显存压力，为卸载更多层留出空间。

---

## 7. 注意力优化

### `-fa on`
- **作用**：启用 Flash Attention。
- **调试原因**：
  - Flash Attention 是一种高效注意力实现，能减少显存读写，加速注意力计算。
  - 开启后，显存占用进一步降低，推理速度提升约 5-10%，且不影响生成质量。
  - 对于长上下文尤其有利。

---

## 8. 其他注意事项

- **内存带宽**：最终速度受限于 CPU 内存带宽。本配置在双通道 DDR5-5600 下达到 11 t/s；若升级到更高频率或四通道内存，速度可进一步提升。
- **模型量化选择**：`Q4_K_XL` 是高质量 4-bit 版本，若追求极致速度可尝试 `IQ4_XS`（文件小 10GB），理论上可提升至 12-13 t/s。
- **llama.cpp 版本**：本配置基于较新版本（含 `ggml-cpu-alderlake.dll`），建议定期更新以获得性能改进。

---

## 总结

此配置通过**精细分工**（GPU 处理非专家层 + CPU 处理专家层）和**线程绑定优化**，在 24GB 显存 + 96GB 内存的消费级硬件上，成功将 122B MoE 模型的推理速度提升至 **11 token/s**，实现了流畅的交互体验。每个参数都经过实测调整，兼顾性能与稳定性，可作为类似配置的参考模板。
